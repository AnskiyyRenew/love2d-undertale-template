    开发过程的注意事项¶

    由于模板不是你们写的，所以我们会对你提供帮助，但是我们要帮助的人也是有限度的。

    如何正确的提问¶

    提问前应做的准备¶

    • 先自行搜索：文档（就是这里）、FAQ、Issue（在 Github 上）、Stack Overflow、搜索引擎，确认不是已知问题或重复提问。
    • 尝试简化问题：定位到最小受影响范围（模块、文件、配置），能否在本地复现。

    如何描述问题（必备信息）¶

    • 环境信息：操作系统、语言/运行时版本、依赖或框架版本。
    • 期望行为 vs 实际行为：清晰表达你希望得到的结果和当前得到的结果。
    • 错误信息：完整错误堆栈或日志（不要只截取一行），标注出现时刻或触发条件。
    • 重现步骤：按顺序列出能稳定触发问题的最少步骤，最好能被他人复制执行。

    提供可执行的最小复现示例¶

    • 给出可运行的最小代码片段或最小仓库链接（删掉无关内容）。
    • 避免分享敏感信息（API Key、密码、私有数据），必要时用占位符替代。
    • 说明如何运行示例：命令、输入、预期输出。

    已尝试的解决办法¶

    • 列出你已尝试过的方案与结果（包括查阅过的文档或贴过的链接）。
    • 说明你排查到的可疑点和你认为可能的根因。

    提问礼仪（提高回答概率）¶

    • 标题简洁且具描述性（包含错误关键字或场景）。
    • 内容结构化、语言简练，使用代码块展示代码和日志。
    • 对帮助者友好：说明你希望得到的帮助类型（解答思路、代码修复、定位建议等）。
    • 遇到解决方案及时反馈结果并关闭/更新问题。

    遵循以上要点可以显著提高获得有效帮助的概率，并让问题更快得到解决。

    代码规范情况¶

    命名规范¶

    • 变量和函数使用 snake_case（蛇形命名法）。
    • 常量使用 UPPER_SNAKE_CASE。
    • 类/模块使用 PascalCase。
    • 避免使用单字母变量名，除非在简短循环中使用 i、j、k。

    代码格式¶

    • 使用 2 或 4 个空格进行缩进（保持项目一致）。
    • 每行代码不要太长。
    • 在逻辑块之间添加空行以提高可读性。
    • 使用 local 关键字声明局部变量，避免污染全局作用域。

    注释和文档¶

    • 为复杂逻辑和公共函数添加注释。
    • 使用 --- 开头的 Lua 文档注释说明函数参数和返回值。
    • 保持注释简洁，避免过度注释。

    函数设计¶

    • 单个函数职责单一。
    • 函数应有明确的返回值说明。

    错误处理¶

    • 提供有意义的错误消息便于调试。
